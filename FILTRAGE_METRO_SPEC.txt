SPEC — Script Python de filtrage "métro uniquement" des fichiers GTFS
=======================================================================

Objectif
--------
Réduire tous les fichiers GTFS du dossier src/main/resources/Datas/ pour ne
conserver que les lignes concernant le réseau métro (route_type = 1), en
respectant les dépendances entre fichiers (un service, un arrêt ou une
agence ne doit être supprimé que s'il n'est plus référencé par aucune
circulation de métro conservée).

Fichiers concernés (emplacement : src/main/resources/Datas/)
--------------------------------------------------------------
- agency.txt
- routes.txt
- trips.txt
- stop_times.txt (ou stop_times_metro.txt s'il n'existe déjà que la
  version filtrée)
- stops.txt
- pathways.txt
- transfers.txt
- stop_extensions.txt
- calendar.txt
- calendar_dates.txt

Principe général
-----------------
Le filtrage doit être fait dans un ORDRE PRÉCIS, car chaque fichier ne peut
être filtré qu'à partir des identifiants (routes, trips, stops, services,
agences) retenus à l'étape précédente. Ne pas filtrer les fichiers de
façon indépendante : reconstruire la chaîne de dépendances.

Chaîne de dépendances GTFS pertinente ici :

    routes.txt (route_type=1)
        -> route_id retenus
    trips.txt (route_id in route_id retenus)
        -> trip_id retenus, service_id retenus
    stop_times.txt (trip_id in trip_id retenus)
        -> stop_id retenus (niveau quai/plateforme)
    stops.txt (stop_id retenus + fermeture transitive des parent_station)
        -> stop_id finaux (quais + stations + entrées/niveaux parents)
    pathways.txt / transfers.txt / stop_extensions.txt
        -> filtrés sur les stop_id finaux
    agency.txt
        -> filtré sur les agency_id des routes retenues
    calendar.txt / calendar_dates.txt
        -> filtrés sur les service_id retenus

Étapes détaillées
------------------

1) routes.txt
   Colonnes : route_id, agency_id, route_short_name, route_long_name,
              route_desc, route_type, route_url, route_color,
              route_text_color, route_sort_order
   - Garder uniquement les lignes où route_type == "1" (métro).
   - Construire l'ensemble ROUTES_METRO = { route_id }
   - Construire l'ensemble AGENCIES_METRO = { agency_id } (agences de ces
     routes)

2) trips.txt
   Colonnes : route_id, service_id, trip_id, trip_headsign,
              trip_short_name, direction_id, block_id, shape_id,
              wheelchair_accessible, bikes_allowed
   - Garder uniquement les lignes où route_id est dans ROUTES_METRO.
   - Construire TRIPS_METRO = { trip_id }
   - Construire SERVICES_METRO = { service_id }

3) stop_times.txt (fichier des horaires, potentiellement volumineux :
   NE PAS charger tout le fichier en mémoire, le traiter ligne par ligne
   en streaming)
   Colonnes attendues (standard GTFS, à vérifier sur le fichier réel) :
   trip_id, arrival_time, departure_time, stop_id, stop_sequence, ...
   - Garder uniquement les lignes où trip_id est dans TRIPS_METRO.
   - Pendant ce même passage, collecter STOPS_DIRECTS = { stop_id }
     (tous les stop_id effectivement desservis par un trip métro).
   - Écrire le résultat en remplacement de/vers stop_times_metro.txt.

4) stops.txt
   Colonnes : stop_id, stop_code, stop_name, stop_desc, stop_lon,
              stop_lat, zone_id, stop_url, location_type, parent_station,
              stop_timezone, level_id, wheelchair_boarding, platform_code
   - Partir de STOPS_DIRECTS (quais desservis par le métro).
   - IMPORTANT : remonter la hiérarchie GTFS via parent_station. Un quai
     (location_type=0) pointe vers une station (location_type=1) via son
     parent_station ; une station peut elle-même avoir un parent (entrée,
     niveau...). Faire une résolution récursive/itérative jusqu'à point
     fixe : pour chaque stop_id retenu, si son parent_station n'est pas
     vide et pas déjà retenu, l'ajouter, et recommencer jusqu'à ce que
     plus aucun ajout ne soit fait.
   - Construire STOPS_FINAUX = STOPS_DIRECTS + toute la fermeture des
     parent_station.
   - Garder uniquement les lignes de stops.txt dont stop_id est dans
     STOPS_FINAUX.

5) pathways.txt
   Colonnes : pathway_id, from_stop_id, to_stop_id, pathway_mode,
              is_bidirectional, length, traversal_time, stair_count,
              max_slope, min_width, signposted_as,
              reversed_signposted_as
   - Garder uniquement les lignes où from_stop_id ET to_stop_id sont
     tous les deux dans STOPS_FINAUX.

6) transfers.txt
   Colonnes : from_stop_id, to_stop_id, transfer_type, min_transfer_time
   - Garder uniquement les lignes où from_stop_id ET to_stop_id sont
     tous les deux dans STOPS_FINAUX.

7) stop_extensions.txt
   Colonnes : object_id, object_system, object_code
   - object_id référence un stop_id.
   - Garder uniquement les lignes où object_id est dans STOPS_FINAUX.

8) agency.txt
   Colonnes : agency_id, agency_name, agency_url, agency_timezone,
              agency_lang, agency_phone, agency_email, agency_fare_url
   - Garder uniquement les lignes où agency_id est dans AGENCIES_METRO.

9) calendar.txt
   Colonnes : service_id, monday, ..., sunday, start_date, end_date
   - Garder uniquement les lignes où service_id est dans SERVICES_METRO.

10) calendar_dates.txt
   Colonnes : service_id, date, exception_type
   - Garder uniquement les lignes où service_id est dans SERVICES_METRO.

Contraintes d'implémentation
-----------------------------
- Utiliser le module csv de Python (pas de split(",") naïf : certains
  champs peuvent être vides ou contenir des séparateurs cités entre
  guillemets).
- Conserver la ligne d'en-tête telle quelle dans chaque fichier de
  sortie.
- Ne jamais écraser les fichiers sources directement en premier essai :
  écrire vers un dossier séparé (ex. Datas_filtered/) ou faire une copie
  de sauvegarde des fichiers originaux avant remplacement, afin de
  pouvoir revenir en arrière en cas d'erreur de filtrage.
- Traiter stop_times.txt en streaming (lecture/écriture ligne par ligne),
  car ce fichier peut être très volumineux (plusieurs centaines de Mo
  dans sa version non filtrée). Les autres fichiers peuvent être chargés
  entièrement en mémoire (ils sont beaucoup plus petits).
- Journaliser pour chaque fichier : nombre de lignes lues / nombre de
  lignes conservées, pour permettre une vérification rapide du filtrage.
- Le critère "route_type == 1" doit rester une constante nommée/
  documentée dans le script (cohérent avec TYPE_METRO=1 utilisé côté
  Java dans ChargeurGTFS.java), au cas où le périmètre évoluerait
  (ex. RER, tram) dans une future itération.

Résultat attendu
----------------
Un jeu de fichiers GTFS ne contenant plus que les données nécessaires au
réseau métro (routes, circulations, horaires, arrêts, correspondances,
agences, calendriers), directement exploitable par ChargeurGTFS /
ConstructeurGraphe sans filtrage supplémentaire côté Java.
